## Why Runjucks?

<CardGrid stagger>
	<Card title="Nunjucks-shaped API" icon="star">
		`renderString` and `Environment` mirror the API you know from [Nunjucks](https://mozilla.github.io/nunjucks/).
	</Card>
	<Card title="Rust core" icon="rocket">
		Lex → parse → render in Rust: string-aware delimiters, whitespace control, and expression parsing (nom) — no compile-to-JS `eval` in the hot path.
	</Card>
	<Card title="TypeScript" icon="document">
		Types ship with the package (`index.d.ts`). Full reference is [generated on build](api/).
	</Card>
	<Card title="Conformance" icon="approve-check">
		Golden JSON fixtures under `native/fixtures/conformance/`; see [Development](guides/development/) and [Syntax and parity](guides/syntax/) for what is tested and what is skipped.
	</Card>
</CardGrid>

### Status

**Working today:** plain text, `{# #}` comments, `{{ … }}` with literals, operators, `is` / `in`, and Nunjucks-style trim markers; the **lexer** emits `{% … %}` tags (and `raw` / `verbatim` blocks).

**Not wired through render yet:** template-level [`parse()`](rustdoc/runjucks_core/parser/fn.parse.html) still **errors on `Token::Tag`** — so full `{% if %}`, `{% for %}`, filters (`|`), and loaders are **not** end-to-end like upstream Nunjucks. See [Syntax and parity](guides/syntax/).
